# Internationalization Lens

Active when the project's **characteristics** include being multilingual —
serving more than one language or locale. i18n done late is a rewrite: hardcoded
strings, layouts that break on longer translations, and date/number formats baked
to one locale are cheap to prevent and expensive to retrofit. This lens keeps the
product translatable and locale-correct as it's built.

Like every lens, it flags concerns and proposes NFRs; it never invents scope —
which locales are in scope is the PO's call, not the lens's.

## Who owns what

| Area | Owner phase | Agent |
|---|---|---|
| Locale scope & formatting expectations in the spec | Specify | Product Owner |
| Layout under translation & RTL in the spec/mockup | Specify (design-check) | Designer |
| No hardcoded strings, locale-aware formatting in the diff | Review | Reviewer |
| Rendered locales, formats, RTL in a real browser | QA | QA Tester |

## The checklist

### 1. Externalized text *(Review)*
- [ ] **No user-facing string is hardcoded** — all text goes through the i18n layer with keys; concatenated sentence fragments are avoided (they don't translate).
- [ ] Pluralization and gender use the i18n library's rules, not `count === 1 ? 'item' : 'items'` hand-rolled per language.
- [ ] Translation keys for the feature exist for every in-scope locale; a missing key has a defined fallback, not a blank or the raw key shown to users.

### 2. Locale-aware formatting *(Review, verified in QA)*
- [ ] Dates, times, numbers, currencies and units are formatted through locale-aware APIs — never a hardcoded `MM/DD/YYYY` or `$` prefix.
- [ ] Time zones are handled explicitly; "today" is the user's today, not the server's.
- [ ] Sorting and collation respect the locale where it matters.

### 3. Layout & direction *(Design, verified in QA)*
- [ ] Layout survives **text expansion** — German/Finnish run ~30% longer than English; buttons and labels don't clip or overflow.
- [ ] **RTL** locales (if in scope) mirror layout, icons and flow correctly; no left-anchored assumptions.
- [ ] `<html lang>` (and `dir`) is set per active locale; the language switcher is discoverable and persists the choice.

### 4. Content & assets *(Review/Design)*
- [ ] Images with embedded text, and locale-specific assets, have a localization path — not one baked-in-English graphic for all locales.
- [ ] Untranslated placeholder text (lorem ipsum, English stand-ins) never ships to a non-default locale.

## Phase notes

- **Specify (PO):** state the in-scope locales and formatting expectations as
  ACs/NFRs — e.g. "UI available in de/en/fr", "dates render in the active
  locale", "layout holds at +30% text length", "RTL supported for ar". Traceable
  downstream.
- **Design (Designer):** check the mockup/spec for text-expansion tolerance and,
  if RTL is in scope, mirroring. These are `Major`/`Minor` findings with location
  and fix.
- **Review (Reviewer):** trace externalized strings and locale-aware formatting
  in the diff. A hardcoded string or a `toLocaleString`-less date is a finding
  tied to its NFR.
- **QA (QA Tester):** switch locales in the browser and observe — translated
  strings (no raw keys), correct formats, RTL layout intact, nothing clipped.
  Report per `NFR-n`.
